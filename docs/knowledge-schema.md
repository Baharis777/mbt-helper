# Схема базы знаний

Этот документ описывает первую структуру файлов базы знаний для ассистента.

## Языковые правила

Техническая структура остается на английском:

- Названия файлов.
- Названия папок.
- TypeScript-типы.
- Имена функций.
- Имена полей.
- Названия documentation-файлов.

Контент базы знаний для менеджеров должен быть написан на русском, если исходные данные не доступны только на английском.

Русскоязычный контент включает:

- Карточки туров.
- Tone of voice продаж.
- Политики.
- Примеры разговоров.
- Примеры ответов клиентам.
- Предупреждения для менеджеров.

Если используется англоязычный источник, нужно сохранить смысл и по возможности добавить русскоязычное резюме или пометку на проверку. Английские ответы клиентам могут поддерживаться позже, но русский остается языком внутренней базы знаний по умолчанию.

## Активный объем

Активная база MVP сейчас содержит только Mount Batur. Небатурные туры, включая Убуд, Улувату и обзорные туры по Бали, должны находиться в архиве и не использоваться как активное знание.

## Структура Batur sunrise

Семейство рассветного треккинга на Батур хранится в `knowledge/tours/batur-sunrise/`.

- `page.md`: только информация, общая для всех вариантов семейства.
- `variants/`: только факты, специфичные для конкретного варианта.
- Варианты с якорными ссылками используют временные ID, пока точные названия не подтверждены.

Активная структура:

```text
knowledge/tours/batur-sunrise/
  page.md
  variants/
    batur-variant-1906.md
    batur-variant-2095.md
    batur-variant-1728.md
    batur-variant-1008.md
    batur-variant-1905.md
    batur-variant-1009.md
    batur-variant-2094.md
```

## Поля тура

- `title`: название тура для клиента. Значение по умолчанию пишется на русском, если официальное название продукта не закреплено на английском.
- `category`: категория тура. Технические значения категорий могут оставаться на английском.
- `status`: статус карточки, например `draft`, `active` или `archived`.
- `audience`: подходящий профиль клиента, на русском.
- `summary`: короткое внутреннее описание, на русском.
- `highlights`: ключевые аргументы продажи, на русском.
- `itinerary`: порядок программы, на русском.
- `inclusions`: что включено в тур, на русском.
- `exclusions`: что не включено, на русском.
- `pickup`: районы и правила трансфера, на русском.
- `duration`: ожидаемая длительность, на русском.
- `difficulty`: физическая нагрузка и доступность, на русском.
- `what_to_bring`: что клиенту подготовить, на русском.
- `policies`: ссылки или отсылки к связанным policy-файлам.
- `sales_notes`: позиционирование, возражения и апсейл, на русском.
- `manager_notes`: внутренние операционные заметки и предупреждения менеджеру, на русском.

## Разделение семейства и вариантов

Файл семейства `page.md` должен содержать только то, что действительно общее для всех вариантов Batur sunrise. Если факт может отличаться между вариантами, он должен быть перенесен в соответствующий файл `variants/`.

Файлы вариантов должны содержать только факты конкретного варианта:

- Точное название варианта.
- Цена.
- Продолжительность.
- Время выезда.
- Что включено.
- Что не включено.
- Точная программа.
- Правила отмены.
- Доступность.
- Статус безопасности.
- Ограничения по возрасту, здоровью и физической нагрузке.

Если факт не подтвержден, нужно писать `NEEDS_REVIEW`.

## Источники

Файл `knowledge/sources.md` использует структуру:

```text
| item_id | item_type | parent_id | source_url | source_type | status | notes |
```

Допустимые значения `item_type`:

- `tour_family`
- `tour_variant`
- `policy`
- `live_status`

Для вариантов `parent_id` должен быть `batur-sunrise`.

Для главной страницы Batur sunrise `source_type` должен быть `website`. Для anchor-вариантов `source_type` должен быть `website_anchor`.

`status` должен быть `needs_import` или `needs_review`.

## Обязательная ручная проверка

Любой вопрос про извержение, активность вулкана, пепел, дым, лаву, землетрясения, закрытие маршрута, текущую безопасность или доступность должен требовать ручной проверки перед подтверждением клиенту.

## Политики

Файлы политик должны находиться в `knowledge/policies/`. В активном MVP используются только политики, применимые к Mount Batur.

Рекомендуемые темы:

- Отмена и перенос бронирования.
- Погода и безопасность.
- Трансфер и опоздания.
- Минимальный возраст или требования к физической форме.
- Оплата и депозит.

## Продажи

Рекомендации по продажам должны находиться в `knowledge/sales/`.

Начальный файл — `tone-of-voice.md`. Он на русском описывает, как ассистент должен формулировать черновики ответов клиентам.

## Примеры разговоров

Активные Batur-примеры разговоров должны находиться в `examples/conversations/`. Небатурные примеры должны храниться в `examples/_archive/non-batur/`.

Используйте примеры, чтобы фиксировать:

- Частые вопросы менеджеров.
- Желаемую форму ответа.
- Примеры клиентских ответов.
- Предупреждения для менеджеров.
- Ситуации эскалации, где ассистент не должен угадывать.
